# Agent Instructions

## Mission

Build a concise, reliable prototype named CaseFlow for GoTyme P2: Adaptive AML/KYT workflow engine. Optimize for a five-minute pitch, a 60-second live demo, and a complete AABW submission.

## Product rules

- Treat all customer, transaction, wallet, Travel Rule, policy, and case data as synthetic.
- Present automated outputs as recommendations, never final compliance or customer-impacting actions.
- Use exactly `CLEAR`, `ESCALATE`, and `REQUEST_EVIDENCE`.
- Every decision includes evidence IDs, policy citations, confidence, objections, and next action.
- Human approval is mandatory for overrides and policy activation.
- Learning means a versioned proposal plus historical replay; never silently mutate prompts, allowlists, thresholds, or policy.
- Do not claim metrics that were not generated by the evaluation harness.
- The submission is GoTyme-only. Do not describe CaseFlow as an AWS-track entry.

## Engineering rules

- Prefer typed contracts, schema validation, and deterministic state transitions over free-form agent prose.
- Keep roles bounded: Case Intake, Triage, conditional Investigation, Compliance Challenger, and Orchestrator.
- Cap critique at two rounds and stop safely when evidence remains insufficient.
- Build the smallest vertical slice first: one alert to one cited decision packet.
- Seed demo data and provide reset, cached-response, and local fallback paths.
- Keep expected evaluation labels separate from runtime prompts.
- Do not add live banking, payment, KYC, sanctions, or personal-data integrations.

## Demo rules

- Make goal, plan, tool calls, actions, and verification visible.
- Use the incomplete Travel Rule case as the primary live story.
- Finish the live product walkthrough in 60 seconds.
- End on the decision packet and human-controlled policy proposal, not architecture.
- Avoid a chat-first interface; the primary UI is an analyst case-review workflow.
